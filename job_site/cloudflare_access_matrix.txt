RISK 1 — Access bypass not confirmed on custom domain (all five paths):
  Severity: HIGH for /telegram/webhook, MEDIUM for the others.
  The custom-domain Access Application from Stage 1/2 defaults to
  blocking all paths with a 302. Operator must confirm or add a
  Bypass policy for these five paths before running CF-TEST against
  the custom domain.
  Mitigation: Use *.workers.dev surface for all CF-TEST evidence
  unless operator confirms bypass. workers_dev: true is set; this
  surface is the valid fallback per the build sheet.

RISK 2 — /telegram/webhook high-priority Access bypass:
  Severity: HIGH.
  Telegram cannot send CF-Access-Client-Id / CF-Access-Client-Secret
  headers. If the webhook URL points at the custom domain without a
  confirmed bypass, 100% of inbound updates are silently dropped by
  Cloudflare (Telegram marks them as last_error).
  Mitigation: Keep Telegram webhook registered against the
  *.workers.dev hostname for Stage 4 CF-TEST. Do not change to the
  custom domain unless operator confirms the bypass rule is active
  and tested.

RISK 3 — Access posture split between / and /session/latest:
  Severity: MEDIUM.
  If "/" is Access-bypassed but /session/latest is not (or vice
  versa), the UI loads but the event trail is blank — XHR returns
  Access login HTML instead of JSON. Both paths must share the same
  bypass posture.
  Mitigation: Bypass all five paths together as a single set.

RISK 4 — Calendar provider env vars not set in production:
  Severity: LOW (runtime env item, not an Access item).
  GOOGLE_CALENDAR_* vars (or equivalent) may not be present in the
  production secrets. /calendar/status returns 200 with
  provider.ready=false when unconfigured. This is the declared
  "unconfigured" state per Stage 4 design and will not block CF-TEST
  (probe returns 200). Operator should note provider.ready=false in
  CF-TEST evidence if calendar env vars are absent.

RISK 5 — Manual-version naming drift (minor, pre-existing):
