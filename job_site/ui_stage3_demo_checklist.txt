1. STAGE 3 S5 CONTRACT
----------------------------------------------------------
S5 goal: "Make the UI clearly show processing state for text, voice,
and calendar actions without redesigning the shell."

Pinned by this checklist:

  U1. Status bar shows Calendar provider readiness.
      #status-calendar displays "demo ready" (green) when
      CALENDAR_PROVIDER=demo, "not ready" (red) otherwise.
      Element updates on every poll cycle without page reload.

  U2. Voice events are clearly labelled in the transcript pane.
      Inbound items with kind="voice" render a "🎙 VOICE" badge
      above the body text. When text is null (pre-transcription),
      the body shows "[voice note — transcription pending]" rather
      than a raw "[voice]" token.

  U3. Calendar action result bar appears after the first ?run invocation.
      #calendar-action-bar is hidden by default and reveals itself
      when /calendar/status returns a non-null invocation block.
      Bar shows: "✓ calendar.list_today ok · N events · HH:MM:SS"
      (green) on success, or "✗ calendar.list_today failed (code) ·
      HH:MM:SS" (red) on failure.

  U4. Bar reflects current invocation on every poll — no stale state.
      Polling the status route every 5 s keeps the bar live without
      page reload.

  U5. Layout is not redesigned.
      The two existing transcript / response panes are unchanged.
      The calendar action bar is an additive strip between the status
      bar and the main panels; it does not alter the pane grid.

6. PROOF STEPS — CALENDAR ACTION RESULT BAR (U3 / U4)
----------------------------------------------------------
Step 1 — confirm bar is hidden before first ?run
  curl -s "https://<worker>.workers.dev/calendar/status" | jq '.invocation'

  Expect: null

  In the operator UI (after poll):
    #calendar-action-bar must not be visible (display: none).

Step 2 — trigger the action via ?run=1
  curl -s "https://<worker>.workers.dev/calendar/status?run=1" \
    | jq '{ok: .invocation.ok, event_count: .invocation.payload.event_count}'

  Expect: { "ok": true, "event_count": 3 }

  In the operator UI (after next poll ≤5 s):
    Calendar action bar appears below the status bar.
    Bar reads: ✓ calendar.list_today ok · 3 events · HH:MM:SS
    Text is rendered in green.

Step 3 — trigger the failure path
  wrangler secret delete CALENDAR_PROVIDER
  curl -s "https://<worker>.workers.dev/calendar/status?run=1" \
    | jq '{ok: .invocation.ok, code: .invocation.error.code}'

  Expect: { "ok": false, "code": "provider_not_ready" }

  In the operator UI (after next poll):
    Bar reads: ✗ calendar.list_today failed (provider_not_ready) · HH:MM:SS
    Text is rendered in red.

  Restore: wrangler secret put CALENDAR_PROVIDER → demo.
