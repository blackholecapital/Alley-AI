SAFETY INVARIANTS (apply to all paths)
----------------------------------------------------------
I-1  Every route returns HTTP 200 on any provider state.
     A non-200 from an operational route is a PATCH condition.

I-2  No route handler throws an uncaught exception.
     All provider failures are caught and returned as structured
     JSON (ok:false, error.code from a closed enum).

I-3  Every failure result includes a human-readable reply_text.
     Users are never shown raw stack traces or empty responses.

I-4  All routes with no required env still return a valid response
     when those env keys are missing. Degraded ≠ broken.

I-5  (Stage 4) Every /calendar/status and /session/latest response
     includes a pre-formatted view block (ui / pipeline) so the UI
     can render without re-deriving state from raw fields.

PATH 3: BROWSER VOICE CAPTURE
----------------------------------------------------------
Route:      POST /voice/capture
Test:       use operator UI mic button to capture and submit audio

READY STATE
  condition:  Browser grants mic permission (HTTPS required)
              TELEGRAM_BOT_TOKEN set (for reply path)
              DEEPGRAM_API_KEY set (for transcription)
  behaviour:  Browser captures audio via MediaRecorder →
              multipart/form-data POST to /voice/capture →
              Worker transcribes via Deepgram → generates reply →
              sends reply to Telegram chat → session store updated;
              /session/latest pipeline updated with received + replied
  browser states: idle → recording → processing → sent

DEGRADED STATE — mic permission denied
  condition:  User denies microphone access in browser
  behaviour:  Client-side only — voice capture button shows error badge
              (#voice-status-badge data-state="error")
              No Worker call is made; no session item recorded
  operator action: check browser microphone permissions; reload page
              after granting access

DEGRADED STATE — HTTP context (not HTTPS)
  condition:  Operator UI served over plain HTTP (wrangler dev without
              HTTPS or non-localhost context)
  behaviour:  navigator.mediaDevices unavailable → voice button hidden
              or badge shows "unsupported"
  operator action: use deployed HTTPS Worker URL for voice demo;
              localhost is treated as secure by most browsers

DEGRADED STATE — no transcription key
  condition:  DEEPGRAM_API_KEY unset or TRANSCRIPTION_PROVIDER=noop
  behaviour:  Audio received by Worker → noop transcription →
              readable failure reply returned → session item recorded;
              pipeline.current_state = "replied" with failure label
  HTTP status: 200

SAFE FALLBACK
  Browser permission denial: no network call, no crash, UI badge shows error.
  Missing transcription key: Worker handles gracefully, records noop reply.
  Route always returns 200 regardless of transcription state.

PROOF REQUIRED (operator)
  Happy path (requires HTTPS + mic permission + DEEPGRAM_API_KEY):
  [ ] Click mic button → recording badge visible → stops on release
  [ ] /voice/capture returns 200, session item recorded
  [ ] /session/latest pipeline shows voice received + replied
  Safe failure (no DEEPGRAM_API_KEY):
  [ ] Click mic button → noop reply received → pipeline shows replied
  Client-side failure (mic denied):
  [ ] Deny mic permission → error badge visible (no network call)

PATH 6: CLOUDFLARE ROUTE VALIDATION
----------------------------------------------------------
Required public routes (not behind Cloudflare Access):
  /health           — worker health probe
  /session/latest   — UI session polling
  /calendar/status  — UI calendar polling
  / (static assets) — operator UI

Routes that MAY be behind Cloudflare Access:
  /telegram/webhook — should NOT be behind Access (Telegram cannot
                      authenticate through CF Access). Must be publicly
                      reachable by Telegram's servers.
  /voice/capture    — browser-accessible endpoint; must be reachable
                      from the operator's browser. May be behind Access
                      if the operator is logged in, but MUST be reachable
                      from the demo browser context.

Validation commands:
  curl -I https://<worker>.workers.dev/health
  Expect: HTTP/1.1 200

  curl -I https://<worker>.workers.dev/session/latest
  Expect: HTTP/1.1 200 (not 302 to Access login)

  curl -I https://<worker>.workers.dev/calendar/status
  Expect: HTTP/1.1 200 (not 302 to Access login)

  curl -I https://<worker>.workers.dev/
  Expect: HTTP/1.1 200 (operator UI assets served)

Method enforcement (all routes):
  GET   /session/latest   → 200
  HEAD  /session/latest   → 200, no body
  POST  /session/latest   → 405 method_not_allowed
  GET   /calendar/status  → 200
  HEAD  /calendar/status  → 200, no body
  POST  /calendar/status  → 405 method_not_allowed

Fallback modes:
  If /session/latest returns 302:
    Check Cloudflare Access settings; add /session/latest to bypass rules.
  If /calendar/status returns 302:
    Same — add to bypass rules.
  If /telegram/webhook gets 302:
    Telegram webhook delivery will silently fail; setWebhook will
    report success but messages never arrive. Remove Access protection
    from /telegram/webhook entirely.
