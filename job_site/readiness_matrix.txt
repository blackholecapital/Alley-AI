SAFETY INVARIANTS (apply to all paths)
----------------------------------------------------------
I-1  Every route returns HTTP 200 on any provider state.
     A non-200 from an operational route is a PATCH condition.

I-2  No route handler throws an uncaught exception.
     All provider failures are caught and returned as structured
     JSON (ok:false, error.code from a closed enum).

I-3  Every failure result includes a human-readable reply_text.
     Users are never shown raw stack traces or empty responses.

I-4  All routes with no required env still return a valid response
     when those env keys are missing. Degraded ≠ broken.

READY STATE
  condition:  TELEGRAM_BOT_TOKEN + TELEGRAM_WEBHOOK_SECRET +
              DEEPGRAM_API_KEY (with TRANSCRIPTION_PROVIDER=deepgram
              or left unset — auto-detected when key is present)
  behaviour:  voice note received → media downloaded → Deepgram
              transcription → text reply sent → UI shows voice inbound
              (🎙 VOICE badge) + text outbound
  proof:      send 5s voice note → bot replies with transcription text

DEGRADED STATE — no transcription key
  condition:  TELEGRAM_BOT_TOKEN + TELEGRAM_WEBHOOK_SECRET set;
              DEEPGRAM_API_KEY unset or TRANSCRIPTION_PROVIDER=noop
  behaviour:  voice note received → inbound item recorded (kind=voice,
              text=null) → handler returns readable failure reply
              ("I heard your voice note but transcription is not
              configured") → outbound item recorded with error text
  HTTP status: 200 (webhook still ACKs; Telegram does not retry)
  UI:         transcript shows voice badge with "[voice note —
              transcription pending]"; response pane shows error text
  operator action: this is a DEMO-SAFE fallback — show it intentionally
              to prove the voice path is wired without needing an API key

SAFE FALLBACK MODE (noop provider)
  Sets TRANSCRIPTION_PROVIDER=noop or leaves DEEPGRAM_API_KEY unset.
  Voice arrives, inbound event is stored, readable error returned.
  Useful for demos where live transcription is not required.
  Voice event is still clearly visible in the UI with kind badge.

PROOF REQUIRED (operator)
  Happy path:
  [ ] Send voice note → Telegram bot replies with transcription text
  [ ] /session/latest shows voice inbound (kind=voice) + text outbound
  [ ] UI transcript pane shows 🎙 VOICE badge on inbound entry
  Degraded path (optional, proves safe failure):
  [ ] Send voice note with no DEEPGRAM_API_KEY set
  [ ] Bot replies with readable error message (not a crash)
  [ ] UI shows 🎙 VOICE badge + "[voice note — transcription pending]"

PATH 3: CALENDAR ACTION (CALENDAR_LIST_TODAY)
----------------------------------------------------------
Route:      GET /calendar/status?run=1
Test:       dispatch CALENDAR_LIST_TODAY action via ?run=1

READY STATE
  condition:  CALENDAR_PROVIDER=demo (no external API key required)
  behaviour:  handler dispatches action → DemoCalendarProvider returns
              3 seed events anchored to today → reply_text formatted
              with bullet list → /calendar/status?run=1 returns
              invocation: { ok:true, event_count:3, reply_text: "..." }
  proof:      curl ?run=1 → ok=true, 3 events, non-empty reply_text

DEGRADED STATE — provider not configured
  condition:  CALENDAR_PROVIDER unset, empty, "none", or "unconfigured"
  behaviour:  /calendar/status returns provider.ready=false,
              provider.reason=provider_not_configured;
              invocation (when ?run=1): ok=false, error.code=provider_not_ready,
              reply_text="I don't have a calendar connected yet. Ask the
              operator to set CALENDAR_PROVIDER and I'll pick it up."
  HTTP status: 200 (route never returns non-200)
  UI:         #status-calendar shows "not ready" in red;
              #calendar-action-bar shows ✗ result after first ?run=1
  operator action: wrangler secret put CALENDAR_PROVIDER → demo

DEGRADED STATE — unsupported provider
  condition:  CALENDAR_PROVIDER set to an unsupported value (e.g. "google")
  behaviour:  provider.ready=false, reason=provider_unsupported;
              invocation ok=false, reply_text references "supported"
  HTTP status: 200
  operator action: change CALENDAR_PROVIDER to "demo" and redeploy

SAFE FALLBACK MODE
  Omit CALENDAR_PROVIDER entirely. The route is always up, always 200.
  The handler returns an explicit not-configured message.
  This is the correct demo of the failure path — no setup needed.

PROOF REQUIRED (operator)
  Happy path:
  [ ] curl /calendar/status?run=1 → invocation.ok=true, event_count=3
  [ ] UI #calendar-action-bar shows ✓ green result
  [ ] UI #status-calendar shows "demo ready" (green)
  Failure path (proves safe failure, zero external dependencies):
  [ ] curl /calendar/status?run=1 with no CALENDAR_PROVIDER
      → invocation.ok=false, error.code=provider_not_ready, HTTP 200
  [ ] UI shows ✗ red result in calendar bar
