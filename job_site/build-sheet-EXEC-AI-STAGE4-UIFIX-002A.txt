8. FAILURE ISOLATION RATIONALE
   8.1 Background: The live deployed UI (ali-ai.xyz-labs.xyz) logs
       "[transcription_failed] provider=deepgram reason=deepgram 401 INVALID_AUTH"
       in the response pane. This is a voice-path-only failure. The /ui/send
       route does not import or call any transcription provider; it imports only
       generateAssistantReply (from telegram-webhook.ts) and the session store
       helpers (recordUiInbound, recordUiOutbound, recordFailure). The failure
       isolation is therefore structural, not configuration-dependent.
   8.2 Test strategy: inject a failure item via recordFailure() directly into the
       session store (as /voice/capture would), then call handleUiSend() and assert
       the response is { ok: true, reply_text: <non-placeholder> }.
   8.3 Expected store state after test: 3 items — kind=failure (pre-injected) +
       kind=text/inbound + kind=text/outbound.
