4. LIVE-PATH OPERATOR DEMO STEPS (post-deploy)
--------------------------------------------------------------------------
Prereqs:
  * Worker deployed from /worker-wb to *.workers.dev (per Stage 3
    /job_site/cloudflare_access_matrix.txt).
  * TELEGRAM_BOT_TOKEN and TELEGRAM_WEBHOOK_SECRET set via
    `wrangler secret put` in the /worker-wb context.
  * Webhook registered against the workers.dev URL OR the custom-domain
    URL with /telegram/webhook in the Access Bypass policy.
  * Operator UI shell open at the Worker root ("/").

Step 1 — pre-flight: confirm /session/latest is empty or stable
  curl -s "https://ali-ai.<account>.workers.dev/session/latest" | jq '.counts'
  Expect: {"inbound":N,"outbound":N,"total":2N} for some N >= 0.
  Record the starting `session_id` from the response.

Step 2 — send the FIRST text message from Telegram
  Action: in the Telegram client, send "first message: how are you" to
          the bot.
  Expect (inside ~2 s):
    - bot reply visible in Telegram client (NOT just a typing indicator,
      NOT silent — actual message text from the bot)
    - operator UI shows one new inbound row AND one new outbound row
      for this turn within one poll interval

Step 3 — verify session store after first turn
  curl -s "https://ali-ai.<account>.workers.dev/session/latest" | jq '.counts'
  Expect: counts.inbound == N+1, counts.outbound == N+1, counts.total == 2N+2
  curl -s "https://ali-ai.<account>.workers.dev/session/latest" | \
    jq '.items[0:2] | map({direction, chat_id, text})'
  Expect: items[0] direction=outbound (newest), items[1] direction=inbound,
          both with chat_id == your private chat id, outbound text non-empty.

Step 4 — send the SECOND text message from Telegram
  Action: in the Telegram client, send "second message: what is on my
          calendar" to the bot.
  Expect (inside ~2 s):
    - second bot reply visible in Telegram client
    - the bot reply must be DIFFERENT in content from the first reply
      (proves reply derives from the message, not a static ack)
    - operator UI shows another in/out pair appended to the trail

Step 5 — verify two-turn interleaving in session store
  curl -s "https://ali-ai.<account>.workers.dev/session/latest" | jq '.counts'
  Expect: counts.inbound == N+2, counts.outbound == N+2, counts.total == 2N+4
  curl -s "https://ali-ai.<account>.workers.dev/session/latest" | \
    jq '.items[0:4] | map({direction, text})'
  Expect (newest-first): outbound, inbound, outbound, inbound.
  When reversed, the chronological order is in:1, out:1, in:2, out:2.

Step 6 — verify session_id stability
  curl -s "https://ali-ai.<account>.workers.dev/session/latest" | jq '.session.session_id'
  Expect: identical to the session_id captured in Step 1.

Step 7 — UI cross-check
  In the operator UI, confirm:
    - both inbound texts appear with the chat user identity
    - both outbound replies appear with the bot identity
    - the order matches Step 5
    - last-event timestamp updates after each turn
