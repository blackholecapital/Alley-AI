4. LIVE-PATH OPERATOR DEMO STEPS (post-deploy, against workers.dev)
--------------------------------------------------------------------------
Prereqs:
  * Worker deployed from /worker-wb to *.workers.dev (per
    /job_site/cloudflare_access_matrix.txt — workers.dev surface).
  * TELEGRAM_BOT_TOKEN and TELEGRAM_WEBHOOK_SECRET set via
    `wrangler secret put` from within /worker-wb.
  * Webhook registered against the workers.dev URL via:
      curl -s -X POST \
        "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/setWebhook" \
        -d "url=https://ali-ai.<account>.workers.dev/telegram/webhook" \
        -d "secret_token=${TELEGRAM_WEBHOOK_SECRET}" \
        -d "allowed_updates=[\"message\"]"
  * Operator UI open at "https://ali-ai.<account>.workers.dev/".

Step 0 — pre-flight: confirm /health responds
  curl -s "https://ali-ai.<account>.workers.dev/health" | jq '.'
  Expect: {"ok": true, "status": "ok", "correlation_id": "..."}
  If this fails, the Worker is not deployed or not reachable.

Step 1 — baseline: record starting session_id and counts
  curl -s "https://ali-ai.<account>.workers.dev/session/latest" | \
    jq '{session_id: .session.session_id, counts: .counts}'
  Record the session_id (S0) and counts (N_in, N_out).

Step 2 — FIRST text exchange
  Action: in the Telegram client, send "schedule a meeting tomorrow at 2pm"
          to the bot.
  Expect (within ~2 s):
    - The bot sends a reply in the Telegram chat thread. The reply must:
        a) NOT be silent (visible text, not just a typing indicator)
        b) NOT start with "received: " (bare receipt = PATCH)
        c) Be assistant-voice text (e.g., references "note it", "schedule it")
    - The operator UI shows a new inbound row AND a new outbound row.

Step 3 — verify COMPLETION after first exchange
  curl -s "https://ali-ai.<account>.workers.dev/session/latest" | \
    jq '.counts'
  Expect: inbound = N_in + 1, outbound = N_out + 1, total = N_in + N_out + 2

  Verify outbound text in session store:
  curl -s "https://ali-ai.<account>.workers.dev/session/latest" | \
    jq '[.items[] | select(.direction=="outbound")] | .[0] | {text, chat_id}'
  Expect: non-empty text string; not "received: ..."; chat_id = your chat id.

  COMPLETION check: if outbound count did not increase, the loop did not close
  — PATCH condition per C2.

Step 4 — SECOND text exchange
  Action: in the Telegram client, send "hello" to the bot.
  Expect (within ~2 s):
    - A DIFFERENT bot reply than step 2 appears in the Telegram thread.
      Same reply for both messages = static ack = PATCH per C4.
    - The operator UI shows a second in/out pair appended to the trail.

Step 5 — verify two-turn interleaving and COMPLETION
  curl -s "https://ali-ai.<account>.workers.dev/session/latest" | \
    jq '.counts'
  Expect: inbound = N_in + 2, outbound = N_out + 2, total = N_in + N_out + 4

  Verify item ordering (newest-first from /session/latest):
  curl -s "https://ali-ai.<account>.workers.dev/session/latest" | \
    jq '[.items[0:4] | .[] | {direction, text}]'
  Expect (newest-first): outbound, inbound, outbound, inbound.
  Reversed (chronological): in:1, out:1, in:2, out:2. (C4)

Step 6 — verify session_id stability
  curl -s "https://ali-ai.<account>.workers.dev/session/latest" | \
    jq '.session.session_id'
  Expect: identical to the session_id S0 captured in step 1.
  If session_id changed, the isolate was recycled mid-demo — note but
  this is a Worker cold-start event, not a code bug.

Step 7 — FAILURE STATE proof step
  Simulate a failed exchange by sending a webhook POST with a wrong secret:
  curl -s -X POST \
    "https://ali-ai.<account>.workers.dev/telegram/webhook" \
    -H "Content-Type: application/json" \
    -H "X-Telegram-Bot-Api-Secret-Token: WRONG-SECRET" \
    -d '{"update_id": 9999999, "message": {"message_id": 999,
         "date": 1745000000, "text": "this should not appear",
         "chat": {"id": 1, "type": "private"},
         "from": {"id": 1, "is_bot": false, "first_name": "t"}}}'
  Expect HTTP response: 200, body: {"ok": true}
  Expect session store: counts UNCHANGED (no inbound, no outbound recorded).

  curl -s "https://ali-ai.<account>.workers.dev/session/latest" | \
    jq '.counts'
  Expect: still N_in + 2, N_out + 2 from step 5 — secret mismatch is a
          silent drop at the route level; no partial state leaks. (C5)

Step 8 — UI cross-check
  In the operator UI, confirm:
    - Both inbound messages appear with the user identity.
    - Both outbound replies appear with distinct text (not identical).
    - The last-event timestamp updates after each successful exchange.
    - No stale or duplicate rows from the failed step 7 attempt.
