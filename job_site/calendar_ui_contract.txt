3. CLOSED UI-STATE ENUM (CalendarUIState)
----------------------------------------------------------
The drawer branches on CalendarUIState, NOT on free-text reasons.
These are the only legal values; the renderer must never read a
provider `reason` string directly.

  unknown       — /calendar/status has not been fetched yet
                  (initial paint before first fetch completes)

  ready         — provider.ready === true
                  Drawer renders provider name + "configured · ready",
                  no guidance, action button enabled, last invocation
                  rendered if present.

  not_ready     — provider.ready === false AND
                  provider.reason is NOT "provider_not_configured"
                  AND is NOT "provider_unsupported"
                  (e.g. upstream_error from an otherwise-named
                  provider). Drawer shows "configured · not ready"
                  and the re-try guidance block.

  unconfigured  — provider.reason === "provider_not_configured"
                  Drawer shows "unconfigured" pill + the demo-mode
                  guidance string naming the wrangler secret to set.

  unsupported   — provider.reason === "provider_unsupported"
                  Drawer shows "unsupported provider" pill + guidance
                  telling operator to set CALENDAR_PROVIDER=demo or
                  leave it empty.

  error         — /calendar/status unreachable, non-200, or malformed
                  Drawer shows "unreachable" pill + the route-error
                  guidance. Action button stays enabled so the
                  operator can retry; a repeat failure renders as a
                  fresh error line.

4. CLASSIFY CONTRACT (pinned)
----------------------------------------------------------
calendar-panel.js exports classifyStatus(data) -> CalendarUIState
with this decision tree:

  data == null || data.provider missing           -> error
  data.provider.ready === true                    -> ready
  data.provider.reason === "provider_not_configured"
                                                  -> unconfigured
  data.provider.reason === "provider_unsupported"
                                                  -> unsupported
  otherwise                                       -> not_ready

This is the sole translation from provider-level
CalendarFailureReason (see /worker-wb/src/integrations/calendar/
provider.ts) into CalendarUIState. The UI never stringifies reason
values directly into pill copy.

7. ACTION TRIGGER PATH (exactly one)
----------------------------------------------------------
User clicks #calendar-panel-run.
  1. Button is disabled.
  2. GET /calendar/status?run=1 is issued (fetch, no-store, JSON).
  3. On 2xx: renderPanelState() is called on the fresh status; then
     renderPanelInvocation(data.invocation) is called to show the
     just-run result.
  4. On 4xx / 5xx / network error: #calendar-panel-error shows
     "Run failed: <message>" and the drawer state pill flips to
     "unreachable" (error). No exception bubbles to the window.
  5. Button is re-enabled.

No other calendar request path exists in the UI. The passive poll
in session-view.js continues to call GET /calendar/status (no run)
every 5 s; the drawer refreshes via its own call on open.

8. UNCONFIGURED vs CONFIGURED VISIBLE DIFFERENCES
----------------------------------------------------------
These are the readable differences an operator sees WITHOUT opening
devtools or reading network traffic:

  configured (ready):
    - #status-calendar reads "<provider> ready" in green
    - #calendar-panel-state pill is green and reads
      "configured · ready"
    - Guidance block is hidden
    - Run button is enabled
    - After one click, result pill is green "ok" with reply_text
      listing today's events and meta "N events"

  unconfigured:
    - #status-calendar reads "<provider> not ready" in red
    - #calendar-panel-state pill is amber and reads "unconfigured"
    - Guidance block is visible with the demo-mode instruction
    - Run button is enabled (fails safely)
    - After one click, result pill is red "fail" with the canned
      reply_text naming CALENDAR_PROVIDER and meta
      "error.code provider_not_ready"

These two surfaces are the "readable unconfigured/configured states"
required by the S4 pass condition.

9. ACCESSIBILITY & RESPONSIVE
----------------------------------------------------------
  - The drawer uses <aside role implied>, aria-hidden toggles with
    open state, aria-label="Calendar settings".
  - The toggle button uses aria-controls="calendar-panel" and
    aria-expanded=true|false.
  - The state pill, reply body, and error line are static text
    nodes — screen readers get the full reply_text on each run.
  - At viewport width <= 640 px the drawer becomes full-screen
    (width: 100vw, border-left removed) so mobile operators can
    still read reply_text without horizontal scrolling.
  - No keyboard-trap; Tab order reaches the drawer buttons and
    returns to the page. Escape-to-close is deliberately NOT
    added here to keep S4 scope minimal; may be layered in S5.
