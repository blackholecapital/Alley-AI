1. SAFETY INVARIANTS (every failure mode obeys all four)
----------------------------------------------------------
I-1  Webhook MUST ack 200. Telegram retries for ~24 hours on any
     non-2xx; a stuck retry loop hides the real failure from the
     operator. Body is `{"ok":true}`.
I-2  Top-level fetch handler MUST NOT throw. Provider exceptions,
     network errors, and malformed payloads are caught and converted
     to failure events. Uncaught throws crash the isolate and emit
     opaque 1101 to Telegram.
I-3  Exactly ONE failure event is appended to the session store per
     failed update. Duplicate emits inflate the UI history.
I-4  The failure event carries a STABLE `failure_reason` from the
     closed enum in §2 so the UI can branch on reason, not free text.
     `failure_detail` carries the human-readable context.

Stage 3 ADDITION:
I-5  Failure paths MUST NOT fabricate an outbound reply. UI surfaces
     the failure badge; operator decides whether to ask the user to
     retry. Audit trail must distinguish "voice failed silently" from
     "assistant responded with apology".

----------------------------------------------------------
2. CLOSED FAILURE-REASON ENUM (operator-facing)
----------------------------------------------------------
    transcription_failed   — provider returned a non-2xx, non-retryable
                             error, or an unparseable response body.
    transcription_timeout  — provider exceeded TRANSCRIPTION_TIMEOUT_MS
                             (default 15 000 ms — inside Telegram's
                             30 s webhook ack window).
    transcription_empty    — provider returned 200 but no text (silence,
                             unknown language, corrupted audio).
    unsupported_media      — update has no voice and no text; e.g.
                             sticker, photo, video, video_note,
                             document, animation. Ack only, no work.
    payload_too_large      — voice.file_size exceeds
                             VOICE_PAYLOAD_LIMIT_BYTES (20 MiB cap).
                             Gate MUST fire before any network fetch.
    media_fetch_failed     — getFile or file-download URL returned
                             non-2xx, threw, or returned the wrong
                             content type.

The current voice.ts uses a finer-grained internal enum
(`getfile_http_error`, `download_network_error`, `duration_too_long`,
`size_too_large`, `getfile_missing_path`, ...). Worker A's S3 task is
to COLLAPSE those internal codes onto the operator-facing enum above
before the failure reaches the session store. The companion test
pins the operator-facing enum.

----------------------------------------------------------
3. CLOSED REPLY-MODE ENUM (Stage 3 addition)
----------------------------------------------------------
    text_fallback   — voice in -> text out (Stage 3 default).
                      Outbound row has kind="text" and
                      reply_mode="text_fallback".
    voice_native    — voice in -> spoken-audio out. OUT OF SCOPE for
                      Stage 3; reserved for a future stage. The
                      operator UI must tolerate the field but no
                      code path emits this value yet.

Build-sheet operator note that justifies Stage 3 default:
  "Voice can close with text fallback if spoken-audio reply is not
   stable enough for this stage."

4. FALLBACK CONTRACT (expected normalized failure event)
----------------------------------------------------------
Every failure emits ONE event conforming to this shape (mirrors the
S5 UI contract, plus the two voice fields):

    {
      "id":             "telegram:<update_id>",
      "source":         "telegram",
      "kind":           "voice",
      "direction":      "inbound",
      "chat_id":        <number>,
      "message_id":     <number>,
      "user_id":        <number | null>,
      "username":       <string | null>,
      "text":           null,
      "received_at":    "<ISO8601>",
      "failure_reason": "<enum value from §2>",
      "failure_detail": "<short human string; never a stack trace>"
    }

The UI shows the failure badge when `failure_reason` is present.
`text` is null on failure so the transcript pane does not render
an empty bubble.

Successful turns emit TWO events (inbound + outbound):

    inbound:
      kind="voice", direction="inbound",
      text=<transcript>, no failure_reason

    outbound:
      kind="text", direction="outbound",
      reply_mode="text_fallback",
      chat_id == inbound.chat.id,
      reply_to_message_id == inbound.message_id,
      derived_from_event_id == inbound.id,
      text non-empty AND derived from the transcript

Mode: transcription_failed

                    returns a body the provider adapter cannot parse
  gate location:    /worker-wb/src/providers/transcription/index.ts

                    failure_reason="transcription_failed"
  reply emitted?:   NO (per I-5)
  operator UI:      response pane shows failure badge + detail text
  test:             "transcription_failed: 200, no throw, one failure
                     event, no outbound"

Mode: transcription_timeout
  trigger:          provider call exceeds TRANSCRIPTION_TIMEOUT_MS
  gate location:    voice.ts wraps the adapter call in an
                    AbortController with a 15 s deadline (Worker A
                    S3 owes this wrapper if not yet present)
  webhook response: 200 {"ok":true}
  session event:    failure_reason="transcription_timeout"
  reply emitted?:   NO
  operator UI:      failure badge "timed out"
  test:             covered by enum closure + safety invariants

Mode: transcription_empty
  trigger:          provider returns 200 with empty/whitespace text
  gate location:    DeepgramTranscriptionProvider.transcribe already
                    returns ok:false reason:"empty transcript" — Worker

                    the failure reaches the session store
  webhook response: 200 {"ok":true}
  session event:    failure_reason="transcription_empty"
  reply emitted?:   NO
  operator UI:      failure badge "no speech detected"

  gate location:    classifier in inbound.ts / voice.ts — runs BEFORE

                    failure_detail="update has no voice and no text"
  reply emitted?:   NO
  operator UI:      failure badge "unsupported media type"
  test:             "unsupported_media: 200, no throw, one failure
                     event, no fetch, no transcribe"

Mode: payload_too_large
  trigger:          message.voice.file_size > VOICE_PAYLOAD_LIMIT_BYTES
  gate location:    voice.ts size gate — runs BEFORE any getFile call
                    (already present today: see voice.ts size_too_large
                    branch; Worker A must surface it as
                    "payload_too_large" to the session store)
  webhook response: 200 {"ok":true}
  session event:    failure_reason="payload_too_large",
                    failure_detail="file_size <N> exceeds limit <LIMIT>"
  reply emitted?:   NO
  operator UI:      failure badge "payload too large"

Mode: media_fetch_failed
  trigger:          getFile non-2xx / unparseable / missing file_path,

  gate location:    voice.ts wraps both fetch() calls; the internal

                    `getfile_network_error`, `download_network_error`,

  webhook response: 200 {"ok":true}
  session event:    failure_reason="media_fetch_failed"
  reply emitted?:   NO
  operator UI:      failure badge "could not fetch media"
  test:             "media_fetch_failed: 200, no throw, one failure
                     event, no outbound"

6. FAILURE-ORDER GATES (decision flow inside voice.ts)
----------------------------------------------------------
Worker A's voice path must evaluate gates in THIS ORDER, because each
earlier gate short-circuits work that later gates would have wasted:

  1. unsupported_media           — no voice field at all
  2. payload_too_large           — voice.file_size > limit (size gate)
  3. media_fetch_failed          — getFile
  4. media_fetch_failed          — download file_path
  5. transcription_timeout       — racing the provider
  6. transcription_failed        — provider non-2xx / parse error
  7. transcription_empty         — provider ok but empty text
  8. SUCCESS                     — emit inbound voice event with text
                                   AND outbound text-fallback reply

----------------------------------------------------------
7. REPLY FALLBACK CONTRACT (Stage 3 expansion)
----------------------------------------------------------
On the SUCCESS branch (gate 8) the webhook MUST also emit one
outbound row with:

  kind                   = "text"
  direction              = "outbound"
  source                 = "telegram"
  reply_mode             = "text_fallback"
  chat_id                = inbound.chat.id
  reply_to_message_id    = inbound.message_id
  derived_from_event_id  = inbound.id
  text                   = non-empty string derived from the
                           transcript (NOT a static ack; identical
                           transcripts may produce identical replies,
                           but distinct transcripts MUST produce
                           distinct replies)

On every failure branch (gates 1-7) NO outbound row is emitted.
This split is the audit-trail invariant the operator UI relies on
to distinguish "we replied" from "we surfaced a failure".
