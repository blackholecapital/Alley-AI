4. PATH B — BROWSER VOICE ROUND TRIP
----------------------------------------------------------
Validates build-sheet validation_check_2.

Setup:
  Same browser / UI. Grant microphone permission ahead of time
  in the browser's site settings (or let the prompt appear on
  first Talk click).

Steps:

B-1  In the voice footer click "🎙 Talk".
     Expected badge transitions:
       idle -> requesting -> recording
     On permission prompt grant access. Badge pulses red during
     recording. #voice-message reads:
       "Recording — press Stop & Send when done."

B-2  Speak a short phrase (3-5 s):
       "calendar check for today please"

B-3  Click "■ Stop & Send".
     Expected badge transitions:
       recording -> processing -> sent
     #voice-message flips to "Audio received by assistant."
     (green) when upload ok.

B-4  Within 15 s the inbound transcript pane receives a voice
     entry with:
       [ ] kind badge "🎙 voice"
       [ ] transcribed text (or "[voice note — transcription
           pending]" followed by a second update with text)

B-5  The outbound pane receives the completed assistant reply
     (should reference today's calendar because CALENDAR_PROVIDER
     is "demo"; otherwise a generic reply is acceptable).

B-6  Probe /session/latest and confirm at least one event has
     kind "voice":
       curl -s "$BASE/session/latest" \
         | jq '[.events[] | select(.kind=="voice")] | length'
     Expect: >= 1

Evidence produced: see EV-B in demo_evidence_requirements.txt.

Denied / unsupported fallbacks (acceptable alternate proof):
  - If the browser blocks the mic, the badge must show "denied"
    and #voice-message explains how to re-grant permission.
  - If the browser lacks MediaRecorder, the badge must show
    "unsupported" and the Talk button stays disabled.
  Either fallback is a valid Stage 4 proof for validation_check_2
  if the primary round trip cannot run on the operator device —
  BUT at least one successful round trip must exist on any
  supported device in the final evidence bundle.

5. PATH C — CALENDAR ACTION (HAPPY + SAFE FAILURE)
----------------------------------------------------------
Validates build-sheet validation_check_3.

C-1 (happy path) — ensure CALENDAR_PROVIDER="demo" is set.
    In the operator UI click the "⚙ Calendar" pill in the
    status bar to open the calendar drawer.
    Confirm:
      [ ] provider name reads "demo"
      [ ] state pill reads "configured · ready" (green)
      [ ] guidance block is hidden
      [ ] Run button is enabled

C-2 Click "Run calendar.list_today".
    Within 3 s the Last Result surface shows:
      [ ] ok pill (green)
      [ ] reply_text listing today's 3 demo events
          (Team standup / Product review / 1:1 with Alex)
      [ ] meta line: "handled_at <ISO> · 3 events"

C-3 Confirm via curl:
      curl -s "$BASE/calendar/status?run=1" \
        | jq '{ok: .invocation.ok,
               reply_text: .invocation.reply_text,
               event_count: .invocation.payload.event_count}'
    Expect: ok=true, event_count=3, reply_text non-empty.

C-4 (safe failure) — set CALENDAR_PROVIDER empty:
      wrangler secret put CALENDAR_PROVIDER   (value: "")
      (Or delete it: wrangler secret delete CALENDAR_PROVIDER.)
    Reload the UI. In the calendar drawer confirm:
      [ ] state pill reads "unconfigured" (amber)
      [ ] guidance block is visible with the demo-mode instruction
      [ ] Run button still enabled

C-5 Click "Run calendar.list_today" again.
    Last Result surface now shows:
      [ ] fail pill (red)
      [ ] reply_text naming CALENDAR_PROVIDER
      [ ] meta line includes "error.code provider_not_ready"
    HTTP status remains 200 (route never throws).

C-6 Restore CALENDAR_PROVIDER="demo" for the final evidence
    capture in C-2. The Last Result surface will re-populate to
    ok on next click.

Evidence produced: see EV-C in demo_evidence_requirements.txt.
