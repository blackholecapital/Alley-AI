1. WHAT "UI PATH" MEANS IN STAGE 4
----------------------------------------------------------------------
The browser operator UI (/apps/operator-shell/src/ui/index.html) renders
the assistant exchange by polling /session/latest every 5 s
(session-view.js POLL_MS = 5000). It reads the shared session-store ring
buffer and paints:

  - Inbound items   → #transcript-log pane ("Transcript (inbound)")
  - Outbound items  → #response-log   pane ("Assistant Response (outbound)")
  - Failure items   → surfaced via pipeline.current_state = 'failed' in
                      /session/latest (session-latest.ts buildPipelineView)

Any write to the session store — regardless of source — surfaces in the
UI on the next poll. The UI path under validation is therefore:

  browser → /session/latest poll → render outbound item text

and the question is whether the outbound text rendered is a COMPLETED
assistant-voice reply (per S2 contract C1) or a bare "received: ..."
receipt (per Stage 3 placeholder).

           lift to /worker-wb/src/lib/assistant.ts). Today's UI does
           not hit /ui/send, so this is a latent drift rather than a
           live regression; flagged here to prevent it from shipping
           unnoticed.

    File:  /apps/operator-shell/src/ui/session-view.js line 29
    Issue: SEND_ENDPOINT = '/telegram/webhook' — a browser POST to this
           route has no Telegram secret header and will be dropped by
           processTelegramUpdate (failure mode §1 in text_failure_modes).
           sendMessage is currently not wired to any visible control in
           index.html, so it is dormant, but if a future text-input is
           added it must point at /ui/send (after F1 is fixed) — not at
           /telegram/webhook.
