1. SAFETY INVARIANTS (every failure mode obeys all five)
----------------------------------------------------------
I-1  /calendar/status ALWAYS returns HTTP 200 regardless of provider
     state. A non-200 here breaks operator monitoring and the UI
     polling loop.

I-2  The action handler MUST NOT throw. Provider errors, missing
     config, and invalid env values are captured and returned as
     ActionResult (ok=false). Uncaught throws would crash the Worker
     isolate.

I-3  Every failure result MUST carry a human-readable reply_text.
     The operator UI renders this directly. Blank, missing, or
     stack-trace reply_text is a PATCH condition.

I-4  Every failure result MUST carry an error.code from the closed
     ActionErrorCode enum so the UI can branch on code, not free
     text. Valid codes:
         "unknown_action" | "provider_not_ready" |
         "provider_error" | "handler_error"

I-5  (Stage 4) Every response — success or failure — MUST carry a
     pre-formatted `ui` block with provider badge, trigger-button
     enablement, and last-result summary. The UI never has to
     re-derive state from raw invocations.

----------------------------------------------------------
2. CLOSED FAILURE-REASON ENUM (provider-facing)
----------------------------------------------------------
These are the values CalendarReadResult.reason carries when ok=false.
Originate in the calendar provider boundary (provider.ts):

  provider_not_configured — CALENDAR_PROVIDER is unset, empty,
                            "none", or "unconfigured". Provider
                            never attempts a network call.

  provider_not_ready      — Provider is named but initialization
                            has not completed (future: OAuth token
                            not yet refreshed, connection pool
                            warming). Reserved for runtime transient
                            states.

  provider_unsupported    — CALENDAR_PROVIDER is set to a name with
                            no registered adapter (e.g. "google",
                            "outlook", "apple"). Stage 4 ships only
                            "demo".

  upstream_error          — Future live adapter completed a network
                            call but the upstream API returned a
                            non-2xx, rate-limit, or quota error.
                            Retryable depends on HTTP status.

  invalid_window          — Computed today-window is invalid (e.g.
                            request.now is NaN). Code or config
                            defect, not a transient state.

----------------------------------------------------------
3. ROUTER-LEVEL FAILURE CODE
----------------------------------------------------------
At the ActionRouter level, one error code can fire before the
provider boundary is reached:

  unknown_action — Dispatched action_type has no registered handler.
                   Stage lock: only CALENDAR_LIST_TODAY is
                   registered, so any other action_type returns
                   unknown_action. Never user-visible in Stage 4 —
                   surfaces in logs and developer tooling.

4. STAGE 4 UI VIEW BLOCK — PER MODE
----------------------------------------------------------
Each failure reason maps deterministically to a `ui` view so the
operator shell renders the same thing every time. Shape:

    ui: {
      provider: { state, label, detail },
      trigger:  { enabled, action_type, label, hint },
      last_result: { state, summary, handled_at, error_code }
    }

Mode → ui mapping:

  Mode: provider_not_configured   (← PRIMARY SAFE-FAILURE PATH)
    ui.provider.state      = "not_configured"
    ui.provider.label      = "not configured"
    ui.provider.detail     = "CALENDAR_PROVIDER is not set; no
                              calendar backend is wired."
    ui.trigger.enabled     = false
    ui.trigger.action_type = "calendar.list_today"
    ui.trigger.label       = "List today's schedule"
    ui.trigger.hint        = "Set CALENDAR_PROVIDER to enable the
                              calendar action."
    ui.last_result.state     = "failed" (after a trigger attempt)
    ui.last_result.summary   = canned reply_text
    ui.last_result.error_code = "provider_not_ready"

  Mode: provider_unsupported
    ui.provider.state      = "unsupported"
    ui.provider.label      = "<name> (unsupported)"
    ui.provider.detail     = "CALENDAR_PROVIDER=\"<name>\" is not a
                              known provider; expected \"demo\" or
                              empty."
    ui.trigger.enabled     = false
    ui.trigger.hint        = "CALENDAR_PROVIDER is set to an
                              unsupported value."
    ui.last_result.state     = "failed"
    ui.last_result.error_code = "provider_not_ready"

  Mode: provider_not_ready (transient)
    ui.provider.state      = "not_ready"
    ui.provider.label      = "<name> (not ready)"
    ui.trigger.enabled     = false
    ui.trigger.hint        = "Calendar provider is not ready."
    ui.last_result.error_code = "provider_not_ready"

  Mode: invalid_window
    ui.provider.state      = "ready" (provider itself is OK)
    ui.trigger.enabled     = true
    ui.last_result.state     = "failed"
    ui.last_result.error_code = "provider_error"

  Mode: upstream_error (future live provider)
    ui.provider.state      = "ready"
    ui.trigger.enabled     = true
    ui.last_result.state     = "failed"
    ui.last_result.error_code = "provider_error"

  Mode: unknown_action (never user-visible in Stage 4)
    Not exposed via /calendar/status (router is internal). Appears
    in logs and would produce ui.last_result.error_code =
    "unknown_action" if ever surfaced.

     again in a moment."

  provider_unsupported:
    "That calendar provider isn't supported yet. Ask the operator
     to pick a supported CALENDAR_PROVIDER."

  upstream_error:
    "The calendar service returned an error. I've logged it for
     the operator."

  invalid_window:
    "I couldn't read a valid time window for today. I've logged it
     for the operator."

  fallback (unrecognized reason):
    "I couldn't read today's schedule (<reason>)."

6. FAILURE-MODE MATRIX
----------------------------------------------------------

Mode: provider_not_configured   (SAFE FAILURE PATH pinned for demo)
  trigger:       CALENDAR_PROVIDER unset, empty, "none", or
                 "unconfigured"
  gate location: provider.ts → UnconfiguredCalendarProvider
                 (returned by getCalendarProvider factory); handler
                 checks provider.ready before calling
                 readTodaysSchedule
  ActionResult:  ok=false, error.code="provider_not_ready",
                 reply_text = canned "no calendar connected" message
  throws?:       NO
  HTTP status:   200
  UI render:     ui.provider.state="not_configured",
                 ui.trigger.enabled=false,
                 ui.last_result.state="failed" with canned summary;
                 calendar status bar shows "not configured";
                 trigger button disabled with hint naming
                 CALENDAR_PROVIDER
  tests:         "safe failure: GET ?run=1 with no CALENDAR_PROVIDER
                  → 200, invocation.ok=false, no throw"
                 "safe failure: UI view reports not_configured +
                  disabled trigger"

Mode: provider_unsupported
  trigger:       CALENDAR_PROVIDER set to unrecognized value ("google",
                 "outlook", "apple")
  gate location: provider.ts → getCalendarProvider factory returns
                 anonymous not-ready provider with
                 reason="provider_unsupported"
  ActionResult:  ok=false, error.code="provider_not_ready",
                 reply_text = canned "not supported" message
  throws?:       NO
  HTTP status:   200
  UI render:     ui.provider.state="unsupported",
                 ui.trigger.enabled=false
  tests:         "safe failure: unsupported provider → state=unsupported,
                  invocation.ok=false"

Mode: invalid_window
  trigger:       DemoCalendarProvider.readTodaysSchedule called with
                 request.now that is not a valid Date (NaN)
  gate location: provider.ts DemoCalendarProvider.readTodaysSchedule
  CalendarReadResult: ok=false, reason="invalid_window"
  ActionResult:  ok=false, error.code="provider_error",
                 reply_text = canned "invalid time window" message
  throws?:       NO
  HTTP status:   200
  UI render:     ui.provider.state="ready" (provider OK),
                 ui.last_result.state="failed",
                 ui.last_result.error_code="provider_error"

Mode: upstream_error (future)
  trigger:       Future live provider returns non-2xx / parse failure
  gate location: Future live provider adapter
  CalendarReadResult: ok=false, reason="upstream_error"
  ActionResult:  ok=false, error.code="provider_error"
  throws?:       NO — adapter must catch and return ok=false
  HTTP status:   200
  UI render:     ui.last_result.state="failed" with canned summary

Mode: unknown_action (router level)
  trigger:       Caller dispatches action_type with no registered
                 handler (e.g. "calendar.create")
  gate location: action-router.ts ActionRouter.dispatch
  ActionResult:  ok=false, error.code="unknown_action"
  throws?:       NO
  HTTP status:   200 (not usually user-visible; logs only)
  tests:         "failure: unknown action type returns ok=false with
                  error.code=unknown_action" (Stage 3 file)

7. FAILURE-ORDER GATES (decision flow inside the handler)
----------------------------------------------------------
buildCalendarListTodayHandler evaluates in strict order:

  1. provider.ready check — fires BEFORE readTodaysSchedule call.
     Returns ok=false (provider_not_configured /
     provider_not_ready / provider_unsupported) without any I/O.

  2. provider.readTodaysSchedule — only reached when provider.ready=true.
     Can return ok=false with reason invalid_window or upstream_error.

  3. SUCCESS — ok=true with event array; formatScheduleReply builds
     reply_text; handler returns ok=true.

Gate 1 always runs first so no network I/O is wasted on a
misconfigured or missing provider.

Stage 4 tightening: after the handler returns, calendar-status.ts
feeds the result (and the provider status from step 1) into
buildCalendarUiView to compose the UI block. The UI block reflects
the exact gate that fired.

9. SAFE-FAILURE PROPERTIES (summary for operator)
----------------------------------------------------------
  a. Worker isolate NEVER crashes from a calendar action failure.
     The action router wraps all handler calls in try/catch; even a
     buggy provider adapter is caught and returns
     error.code="handler_error".

  b. /calendar/status ALWAYS returns HTTP 200 whether the provider
     is configured, unconfigured, unsupported, or transient-unready.

  c. Every failure emits a human-readable reply_text. Users are
     never shown raw error messages or stack traces.

  d. Every failure emits error.code from the closed ActionErrorCode
     enum so the UI can branch on code, not free text.

  e. The single-action scope lock (Stage 3 C1) ensures no accidental
     write action can be dispatched — the router returns
     unknown_action for any other action_type.

  f. (Stage 4) The ui block shape is identical across success and
     failure; the UI renders the same layout and only swaps text
     + badge state. No conditional rendering branch is required.
