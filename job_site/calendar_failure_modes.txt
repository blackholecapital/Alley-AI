1. SAFETY INVARIANTS (every failure mode obeys all four)
----------------------------------------------------------
I-1  The /calendar/status route MUST return HTTP 200 regardless of
     provider state. A non-200 here breaks operator monitoring and any
     UI polling loop.

I-2  The action handler MUST NOT throw. Provider errors, missing config,
     and invalid env values are captured and returned as ActionResult
     (ok=false). Uncaught throws crash the Worker isolate.

I-3  Every failure result MUST carry a human-readable reply_text. The
     operator UI renders this directly to the user. A blank, missing,
     or stack-trace reply_text is a PATCH condition.

I-4  Every failure result MUST carry an error.code from the closed
     ActionErrorCode enum so the UI can branch on code, not free text.
     Valid codes: "unknown_action" | "provider_not_ready" |
                  "provider_error" | "handler_error"

----------------------------------------------------------
2. CLOSED FAILURE-REASON ENUM (provider-facing, CalendarFailureReason)
----------------------------------------------------------
These are the values that CalendarReadResult.reason may carry when
ok=false. They originate in the calendar provider boundary:

    provider_not_configured — CALENDAR_PROVIDER is unset, empty,
                              "none", or "unconfigured". The provider
                              never attempts a network call.

    provider_not_ready      — Provider is named but has not completed
                              initialization (future: OAuth token not
                              yet refreshed, connection pool warming up).
                              Reserved for runtime transient states.

    provider_unsupported    — CALENDAR_PROVIDER is set to a name that
                              has no registered adapter in the factory
                              (e.g. "google", "outlook", "apple").
                              Stage 3 supports only "demo".

    upstream_error          — Provider adapter completed a network call
                              but the upstream API returned a non-2xx,
                              rate-limit, or quota error. Retryable
                              depends on the HTTP status code.

    invalid_window          — The computed today-window is invalid
                              (e.g. request.now is NaN or the timezone
                              arithmetic overflowed). This is a code
                              or config defect, not a transient state.

----------------------------------------------------------
3. ACTION ROUTER FAILURE CODE
----------------------------------------------------------
At the ActionRouter level, one additional error code is possible
before the provider boundary is ever reached:

    unknown_action          — The dispatched action_type has no
                              registered handler. This fires if code
                              outside the calendar boundary dispatches
                              a second (unregistered) action type.
                              Stage 3 scope: only CALENDAR_LIST_TODAY
                              is registered, so any other action_type
                              returns unknown_action.

----------------------------------------------------------
4. CANNED OPERATOR-FACING REPLY_TEXT BY REASON
----------------------------------------------------------
Each failure reason maps to one specific canned message. These are
the strings the handler emits via FAILURE_REPLY_BY_REASON in index.ts.
The operator UI renders these to the user without modification.

  provider_not_configured:
    "I don't have a calendar connected yet. Ask the operator to set
     CALENDAR_PROVIDER and I'll pick it up."

  provider_not_ready:
    "The calendar connection is set up but not ready right now. Try
     again in a moment."

  provider_unsupported:
    "That calendar provider isn't supported yet. Ask the operator to
     pick a supported CALENDAR_PROVIDER."

  upstream_error:
    "The calendar service returned an error. I've logged it for the
     operator."

  invalid_window:
    "I couldn't read a valid time window for today. I've logged it for
     the operator."

  fallback (unrecognized reason):
    "I couldn't read today's schedule (<reason>)."

5. FAILURE-MODE MATRIX
----------------------------------------------------------

Mode: provider_not_configured
  trigger:       CALENDAR_PROVIDER unset, empty, "none", or
                 "unconfigured"
  gate location: calendar/provider.ts → UnconfiguredCalendarProvider
                 (returned by getCalendarProvider factory);
                 handler checks provider.ready before calling
                 readTodaysSchedule
  ActionResult:  ok=false, error.code="provider_not_ready"
                 reply_text = canned "no calendar connected" message
  throws?:       NO
  HTTP status:   200
  operator UI:   failure badge; canned reply_text shown to user;
                 CALENDAR_PROVIDER env var named explicitly
  test:          "failure: dispatch CALENDAR_LIST_TODAY with
                  unconfigured provider returns ok=false"
                 "failure: unconfigured provider reply_text is the
                  canned human-readable message"
                 "failure: unconfigured provider result has
                  error.code = 'provider_not_ready'"
                 "failure: dispatch does NOT throw when provider is
                  unconfigured"

Mode: provider_unsupported
  trigger:       CALENDAR_PROVIDER set to an unrecognized value (e.g.
                 "google", "outlook", "apple")
  gate location: calendar/provider.ts → getCalendarProvider factory
                 returns an anonymous not-ready object with
                 reason="provider_unsupported"
  ActionResult:  ok=false, error.code="provider_not_ready"
                 reply_text = canned "not supported yet" message
  throws?:       NO
  HTTP status:   200
  operator UI:   failure badge; user informed provider is unsupported
  test:          "failure: unsupported provider name returns ok=false
                  with provider_unsupported reason"

Mode: invalid_window
  trigger:       DemoCalendarProvider.readTodaysSchedule called with
                 request.now that is not a valid Date (NaN)
  gate location: calendar/provider.ts DemoCalendarProvider.readTodaysSchedule
  CalendarReadResult: ok=false, reason="invalid_window"
  ActionResult:  ok=false, error.code="provider_error"
                 reply_text = canned "invalid time window" message
  throws?:       NO
  HTTP status:   200
  operator UI:   failure badge; operator prompted to check clock/TZ config

Mode: upstream_error
  trigger:       Future live provider adapter returns non-2xx / parse
                 failure from the calendar API
  gate location: Future live provider adapter (out of Stage 3 scope;
                 demo provider never produces this)
  CalendarReadResult: ok=false, reason="upstream_error", retryable per
                 HTTP status
  ActionResult:  ok=false, error.code="provider_error"
                 reply_text = canned "calendar service returned error"
  throws?:       NO — adapter must catch and return ok=false
  HTTP status:   200
  operator UI:   failure badge; error logged for operator inspection

Mode: unknown_action
  trigger:       Caller dispatches an action_type that is not
                 registered in the ActionRouter (e.g. "calendar.create")
  gate location: action-router.ts ActionRouter.dispatch
  ActionResult:  ok=false, error.code="unknown_action"
                 reply_text names the unrecognized action_type
  throws?:       NO
  HTTP status:   200
  operator UI:   not typically user-visible; surfaces in runtime logs
                 and developer tooling
  test:          "failure: unknown action type returns ok=false with
                  error.code=unknown_action"

6. FAILURE-ORDER GATES (decision flow inside the handler)
----------------------------------------------------------
The handler in buildCalendarListTodayHandler evaluates in this order:

  1. provider.ready check — fires BEFORE readTodaysSchedule call.
     Returns ok=false (provider_not_configured / provider_not_ready /
     provider_unsupported) without any I/O.

  2. provider.readTodaysSchedule — only reached when provider.ready=true.
     Can return ok=false with reason invalid_window or upstream_error.

  3. SUCCESS — readTodaysSchedule returned ok=true with event array.
     formatScheduleReply produces the reply_text; handler returns ok=true.

Gate 1 always runs first so no network I/O is wasted on a misconfigured
or missing provider. This matches the voice path gate order principle.

7. SAFE-FAILURE PROPERTIES (summary for operator)
----------------------------------------------------------
  a. The Worker isolate NEVER crashes (no unhandled throw) due to
     calendar action failure. The action router wraps all handler
     calls in try/catch; even a buggy provider adapter is caught and
     returns error.code="handler_error".

  b. /calendar/status ALWAYS returns HTTP 200 whether the provider is
     configured, unconfigured, or unsupported.

  c. Every failure emits a human-readable reply_text. Users are NOT
     shown raw error messages or stack traces.

  d. Every failure emits an error.code from the closed enum so the UI
     can render the appropriate badge without parsing free text.

  e. The single-action scope lock (C1) ensures no accidental write
     actions (create/update/delete) can be dispatched even if a
     client sends an unexpected action_type — the router returns
     unknown_action.
