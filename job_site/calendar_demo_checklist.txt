5. FAILURE-PATH PROOFS (REQUIRED FOR S4 CHECKPOINT)
----------------------------------------------------------
At least ONE failure mode below MUST be exercised live and captured.

5A. Provider not configured (cheapest — no secret needed)

  Setup: ensure CALENDAR_PROVIDER is unset or set to empty/none.
  (If demo is already set: wrangler secret put CALENDAR_PROVIDER
   then leave it blank or set it to "none".)

  curl -s "https://ali-ai.<account>.workers.dev/calendar/status?run=1" \
    | jq '{provider: .provider,
           invocation_ok: .invocation.ok,
           invocation_reply: .invocation.reply_text,
           error_code: .invocation.error.code}'

  Expect:
    provider.ready        = false
    provider.reason       = "provider_not_configured"
    invocation_ok         = false
    error_code            = "provider_not_ready"
    invocation_reply      — human-readable string mentioning
                            "calendar connected" and "CALENDAR_PROVIDER"
    HTTP status           = 200 (route never throws)

5B. Unsupported provider name

  Setup: wrangler secret put CALENDAR_PROVIDER  (value: "google")

  curl -s "https://ali-ai.<account>.workers.dev/calendar/status?run=1" \
    | jq '{provider: .provider, invocation: .invocation}'

  Expect:
    provider.ready        = false
    provider.reason       = "provider_unsupported"
    invocation.ok         = false
    invocation.reply_text — references "supported" (tells operator
                            to use a known provider)
    HTTP status           = 200

  After verifying, restore CALENDAR_PROVIDER to "demo" for the
  happy-path step 4 evidence capture.
