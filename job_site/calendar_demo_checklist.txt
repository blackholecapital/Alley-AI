1. STAGE 4 S4 CONTRACT (tightening surface)
----------------------------------------------------------
Stage 3 locked the single action boundary. Stage 4 tightens it for
UI use:

  T1. /calendar/status response ALWAYS carries a pre-formatted `ui`
      view block:
          ui.provider.state       ∈ ready | not_configured |
                                     unsupported | not_ready
          ui.provider.label       badge string (e.g. "demo (ready)")
          ui.provider.detail      env-facing detail (empty string if none)
          ui.trigger.enabled      boolean — UI button enablement
          ui.trigger.action_type  "calendar.list_today" (scope lock)
          ui.trigger.label        "List today's schedule"
          ui.trigger.hint         operator hint when disabled
          ui.last_result.state    ∈ ok | failed | none
          ui.last_result.summary  one-line UI text
          ui.last_result.handled_at  ISO string | null
          ui.last_result.error_code  ActionErrorCode string | null

  T2. `triggered_by` discriminator on every GET/HEAD response:
          "none"     — passive readiness poll
          "get_run"  — UI trigger via ?run=1

  T3. HTTP method surface (Stage 3 contract preserved):
          GET   /calendar/status         → readiness, no side effects
          GET   /calendar/status?run=1   → UI trigger
          HEAD  /calendar/status         → readiness probe (no body)
          POST/PUT/PATCH/DELETE          → 405 method_not_allowed
                                           (write actions out of scope)

  T4. last_result persists across calls — UI polling can render the
      most recent invocation even when no run parameter was supplied.

  T5. Route NEVER throws. Every failure is a value returned inside
      the 200 body (invocation.ok=false + ui.last_result.state=failed).
