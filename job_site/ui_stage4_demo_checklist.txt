1. STAGE 4 S5 CONTRACT (session/status surface)
----------------------------------------------------------
S5 adds a `pipeline` block to every GET /session/latest response.

Shape:
    pipeline: {
      current_state: "idle" | "received" | "replied" | "failed",
      current_label: string,   // compact one-line summary for UI
      updated_at:   ISO string | null,
      trail: [                 // newest first, same limit as items
        {
          item_id: string,
          state:   "idle" | "received" | "replied" | "failed",
          label:   string,     // compact per-event label
          at:      ISO string,
          kind:    "text" | "voice" | "other" | "failure",
          source:  "telegram" | "ui",
        },
        ...
      ]
    }

State derivation from session items:
  direction=inbound,  any kind              → state "received"
  direction=outbound, kind=text/other       → state "replied"
  direction=outbound, kind=failure          → state "failed"
  no items in store                         → state "idle"

Labels:
  received + text:    'Text: "[first 80 chars of text]"'
  received + voice:   'Voice note received'   (or with transcript)
  replied:            reply text (first 80 chars)
  failed:             'Failed (failure_code): message (first 80 chars)'
  idle:               'No events yet.'

7. COMPACT VISIBILITY CONTRACT (what the UI renders)
----------------------------------------------------------
For each item in pipeline.trail, the UI should render:

  state=received:
    badge  ← "received"  (or "voice received" for kind=voice)
    label  ← item.label (first 80 chars of text, or "Voice note received")
    time   ← formatTime(item.at)

  state=replied:
    badge  ← "replied"
    label  ← item.label (first 80 chars of reply text)
    time   ← formatTime(item.at)

  state=failed:
    badge  ← "failed"
    label  ← item.label ("Failed (error_code): message")
    time   ← formatTime(item.at)

  state=idle (no trail):
    pane   ← "No events yet."

This maps directly to the existing makeEntry() rendering in
session-view.js. The pipeline.trail is a compact pre-formatted
version of the same data already in items[], allowing the UI to
render a status-strip separately from the full event pane if needed.

8. WHAT REMAINS FOR FULL STATE COVERAGE
----------------------------------------------------------
To surface recording/transcribing/processing/action-called in
/session/latest, the following pipeline recording additions are
needed (out of Stage 4 scope, noted here for Stage 5/6):

  a. voice.ts: call recordPipelineStep('transcribing', event_id)
     before the transcription provider call, and
     recordPipelineStep('processing', event_id) before
     generateAssistantReply(). Requires a new SessionItem
     kind='pipeline_step' or a new 'stage' field on SessionItem.

  b. telegram-webhook.ts: call
     recordPipelineStep('processing', event_id) before
     generateAssistantReply() on the text path.

  c. calendar-status.ts / action-router.ts: call
     recordPipelineStep('action-called', event_id, action_type)
     after ActionRouter.dispatch returns ok=true, and cross the
     ActionResult into the session store so it appears alongside
     the exchange that triggered it.

  recording state is browser-side (voice-capture.js) and cannot be
  surfaced via /session/latest regardless of pipeline recording.
