2. INFERRED /session/latest RESPONSE CONTRACT
-------------------------------------------------------------------
Until Worker A ships /worker-wb/src/routes/session-latest.ts and
/job_site/ui_state_contract.txt, the renderer consumes this shape:

    {
      "session_id":     string | null,
      "state":          "live" | "idle" | "error",
      "last_event_at":  string | null,            // ISO 8601
      "events": [
        {
          "id":           "telegram:<update_id>",
          "direction":    "inbound" | "outbound",
          "source":       "telegram",
          "kind":         "text" | "voice" | "other",
          "text":         string | null,
          "chat_id":      number | null,
          "username":     string | null,
          "received_at":  string                   // ISO 8601
        }
      ]
    }

Inferred from:
  - InternalEvent in /worker-wb/src/integrations/telegram/types.ts
    (id, source, kind, chat_id, message_id, user_id, username, text,
    received_at, raw) — shipped by S2 Worker A.
  - Build-sheet S3 scope: "recent inbound and outbound messages,
    session state, and last-event timestamp".

Deltas vs InternalEvent that Worker A's session-latest.ts must satisfy
for the UI to render without code changes:
  - add `direction: "inbound" | "outbound"` (InternalEvent does not
    carry direction today — outbound replies must be written with
    direction = "outbound").
  - expose the surrounding envelope: session_id, state, last_event_at,
    events[]. The renderer expects events[] length ≥ 0 and tolerates a
    missing last_event_at (falls back to "no events yet").
  - `events[*].id` must be stable across polls so the renderer can
    skip redundant redraws.

3. UI STATE MODEL (rendered states)
-------------------------------------------------------------------
loading   first fetch in flight.
          panes show "loading…", marker shows "loading",
          #status-core = "loading…" with data-state="loading".

empty     fetch succeeded, events.length === 0.
          panes show "no inbound messages yet" /
          "no assistant replies yet"; marker shows "no events yet";
          #status-core = "idle" with data-state="empty".

live      fetch succeeded, events.length >= 1.
          panes render inbound / outbound entries; marker shows
          "last event Xs/m/h ago" and auto-ticks every second;
          #status-core = "live" with data-state="live".

error     fetch threw or non-2xx or malformed JSON.
          panes retain prior content; marker shows "error: <reason>";
          #status-core = "error" with data-state="error". Polling
          continues — transient errors self-heal on the next tick.

    3. inject an inbound text event -> transcript pane renders it;

    5. stop the backend -> error state appears; panes retain content.
    6. restart backend -> renderer recovers on next poll.
