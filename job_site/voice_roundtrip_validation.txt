1. UNDER-TEST PATH (browser /voice/capture)
----------------------------------------------------------------------
The browser operator UI captures audio in voice-capture.js, then POSTs
multipart/form-data to /voice/capture (worker-wb/src/integrations/
telegram/voice.ts:394 handleVoiceCapture). The handler:

  1. method/format/MIME/size validation (lines 399-459)
  2. recordUiInbound({ text: <transcript or "[voice: <mime>, <size> bytes]" })
     ← session-store.ts:165, source='ui', kind='text', direction='inbound'
  3. returns 200 { ok: true, session_id }

The UI then polls /session/latest (session-latest.ts handleSessionLatest)
which derives a pipeline view from the newest item:

  itemState(item):
    direction='inbound'                       → 'received'
    direction='outbound', kind='failure'      → 'failed'
    direction='outbound', kind!='failure'     → 'replied'
    no items                                  → 'idle'

So pipeline.current_state is determined entirely by the most recent
session-store item. After a successful /voice/capture call, the most
recent item is the recordUiInbound entry (direction='inbound'),
producing pipeline.current_state = 'received'.

4. PIPELINE-STATE STUCK-AT-"received" — ROOT CAUSE
----------------------------------------------------------------------
Source: /worker-wb/src/integrations/telegram/voice.ts handleVoiceCapture
        (lines 394-511)

handleVoiceCapture writes exactly one session-store item per accepted
request — recordUiInbound (line 500). It never calls:
  - recordUiOutbound (declared at session-store.ts:190 but only consumed
    by /worker-wb/src/routes/ui-send.ts:72 today — text path only)
  - recordFailure  (consumed only by telegram-webhook.ts text/voice
    paths today)
  - the assistant pipeline (generateAssistantReply, used by
    telegram-webhook.ts and injected into handleTelegramVoiceNote)

Result: every /voice/capture call ends with the newest session-store
item being direction='inbound', so buildPipelineView returns
current_state='received' and current_label set to the inbound text.

Comparison — the Telegram voice path (handleTelegramVoiceNote) closes
the loop in BOTH success and clean-failure branches:

  Branch                     recordOutbound called?  Pipeline ends at
  -------------------------  ----------------------  ----------------
  fetch_failed               yes (FETCH_FALLBACK)    'replied'
  transcription_env_invalid  yes (TRANSCRIPTION_     'replied'
                                  UNAVAILABLE)
  transcription_failed       yes (TRANSCRIPTION_     'replied'
                                  FAILED)
  replied (happy path)       yes (assistantPipeline) 'replied'

So the Stage 3 voice contract (R1: "voice in → ONE outbound reply",
R3: "failure paths record exactly one failure event") is met by the
Telegram path but NOT by the browser /voice/capture path. The browser
path leaves an open loop visible in /session/latest as a perpetual
'received' state.

6. RECOMMENDED FIX (for next pass — out of scope for this validation)
----------------------------------------------------------------------
Add a loop-closure step inside handleVoiceCapture, mirroring the
Telegram voice path:

  6.1 Successful transcription:
      const replyText = generateAssistantReply(transcriptText);
      recordUiOutbound({ event_id: <inbound id>, text: replyText });

  6.2 Clean failure (provider not ready or transcription failed):
      recordFailure({
        event_id:        <inbound id>,
        chat_id:         0,                 // UI synthetic chat id
        failure_code:    'voice_transcription_unavailable'
                          | 'voice_transcription_failed',
        failure_message: <reason>
      });
      // Optional: recordUiOutbound with TRANSCRIPTION_UNAVAILABLE_REPLY
      // if a user-visible reply is desired in the UI response pane.

After 6.1: pipeline.current_state advances to 'replied'.
After 6.2: pipeline.current_state advances to 'failed' with a readable
           failure_code in current_label.

Either choice removes the stuck-at-'received' condition. The cleanest
parity move is 6.1 for the success path (so the UI Response pane shows
an assistant-voice acknowledgement of the spoken request) and the
recordFailure variant of 6.2 for the failure path (so the operator can
distinguish "voice received but not understood" from "voice received
and replied to").

A non-trivial refactor of session-store would extend recordFailure to
accept source='ui' (today it hard-codes source='telegram' at
session-store.ts:140); the recommended minimal change is to add a
recordUiFailure helper symmetric with recordUiInbound/recordUiOutbound.
